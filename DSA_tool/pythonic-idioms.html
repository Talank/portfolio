Big-O Refresher + Pythonic Idioms

Every pattern after this page assumes you're fluent in the tools below. Skim fast if you already know them — the point is recognition speed, not novelty.

▸ Why fluency (not just correctness) is the actual skill here Story

At Bletchley Park during WWII, codebreakers didn't re-derive cryptographic theory for every incoming message — they drilled known shortcuts (cribs, common German phrasing, machine wiring quirks) until recognizing them was reflexive, because the volume of intercepted traffic made "figure it out from first principles each time" operationally useless. Turing's team won on throughput, not on any single message being solved more cleverly than a slower team could have managed eventually.

An interview is the same shape of problem at a smaller scale: you don't get credit for reinventing Counter with a hand-rolled dict-and-loop under time pressure, and every second spent doing that is a second not spent on the actual algorithmic idea the interviewer is grading. The idioms below are this course's "cribs" — drill them until they're reflexive, not derived.

▸ One Piece anecdote: Rayleigh's two years on Rusukaina Island Story

After Whitebeard's war, Luffy doesn't get a montage of him slowly figuring out Haki from first principles in the middle of his next fight — that would get him killed. Instead he spends two full years on Rusukaina Island under Silvers Rayleigh, drilling Observation and Armament Haki over and over until they stop being something he thinks about and become something his body just does. When he shows up at Fishman Island and later Dressrosa, he isn't deriving Haki mid-swing; he's spending his attention on reading the actual fight in front of him, because the technique itself is already reflex.

The training island is the point: it's boring, repetitive, and happens before the fight that matters, precisely so none of that repetition has to happen live. That's exactly what this page is — the two years on Rusukaina before the real interview.

Big-O, the fast version

ComplexityTypical source~Max n for <1s
O(1) / O(log n)hash lookup, binary searchanything
O(n)single pass, sliding window, hashing10⁸
O(n log n)sort-based, heap-based10⁶–10⁷
O(n²)nested loop, DP over pairs~10⁴
O(2ⁿ) / O(n!)brute-force subsets/permutations, unpruned backtracking~20–25
In an interview, say the complexity and justify it ("left and right each move forward at most n times, so O(n) amortized despite the nested-looking loop") — the justification is what's actually being graded.

collections — the single highest-leverage import

from collections import Counter, defaultdict, deque

Counter(words)              # frequency map in one call; .most_common(k) for top-k
defaultdict(list)           # groups without checking "if key not in dict"
deque()                     # O(1) appendleft/popleft — a list is O(n) for these
Using a plain list as a queue (list.pop(0)) is O(n) per pop. Reach for deque the moment you need FIFO behavior — this alone silently turns an "O(n) BFS" solution into O(n²) if missed.

Counter — frequency maps without the boilerplate

Anagram / frequency-comparison problems (e.g. "Valid Anagram", "Group Anagrams") all reduce to "build a count map, then compare."

✗ Manual
counts = {}
for ch in s:
    if ch in counts:
        counts[ch] += 1
    else:
        counts[ch] = 1
✓ Idiomatic
from collections import Counter
counts = Counter(s)
counts.most_common(3)   # [(ch, freq), ...] top-3, no manual sort

Counter IS a dict (subclasses it), so every dict method still works — you're only removing the "did I initialize this key yet" bookkeeping. Counter(a) == Counter(b) is also the entire "is anagram" check in one line.

defaultdict — grouping without existence checks

"Group anagrams," "bucket by remainder," "adjacency list for a graph" are all the same shape: build a dict of lists (or sets) keyed by something derived from each element.

✗ Manual
groups = {}
for word in words:
    key = ''.join(sorted(word))
    if key not in groups:
        groups[key] = []
    groups[key].append(word)
✓ Idiomatic
from collections import defaultdict
groups = defaultdict(list)
for word in words:
    groups[''.join(sorted(word))].append(word)

The whole point of defaultdict: accessing a missing key creates it with the factory's default (list(), int(), set()) instead of raising KeyError — one line disappears from every "group by" loop you'll ever write.

heapq — priority queues without a library

import heapq

heap = []
heapq.heappush(heap, val)
smallest = heapq.heappop(heap)          # heapq is always a MIN-heap
heapq.heappush(heap, -val)              # negate to simulate a max-heap
heapq.heapify(existing_list)            # O(n) in-place, cheaper than n pushes
heapq.nlargest(k, iterable)             # top-k in O(n log k), no manual heap needed

Top-K without sorting the whole array

"Find the k largest/smallest" only needs O(n log k), but the naive instinct is to pay O(n log n) sorting everything.

✗ Overkill
def top_k(nums, k):
    return sorted(nums, reverse=True)[:k]   # O(n log n) — sorts everything
✓ Idiomatic
import heapq
def top_k(nums, k):
    return heapq.nlargest(k, nums)          # O(n log k) — only tracks k at a time

Sorting throws away information you never needed: the relative order of everything *outside* the top k. A size-k heap only ever holds k elements, so each of the n pushes costs O(log k), not O(log n) — the gap matters a lot once k is small and n is huge (e.g. "top 10 of a billion logs").

bisect — binary search you don't have to write

import bisect

bisect.bisect_left(sorted_list, x)      # first index where x could be inserted, keeping sort
bisect.insort(sorted_list, x)           # insert into a sorted list, O(n) but no re-sort

bisect over a hand-rolled binary search loop

"Find the first element ≥ x in a sorted array" is one of the most-rewritten loops in interviews, and it's easy to get the off-by-one boundary wrong live.

✗ Hand-rolled, error-prone
lo, hi = 0, len(arr)
while lo < hi:
    mid = (lo + hi) // 2
    if arr[mid] < x:
        lo = mid + 1
    else:
        hi = mid
first_ge_x = lo
✓ Idiomatic
import bisect
first_ge_x = bisect.bisect_left(arr, x)

The hand-rolled version is correct, but it's exactly the kind of code where a tired off-by-one under interview pressure costs you the whole problem. bisect_left/bisect_right already encode the "first ≥ x" vs "first > x" distinction correctly — use the library for the mechanical part, save your live-coding attention for the part of the problem that's actually novel.

itertools — combinatorics in one line

from itertools import combinations, permutations, product, accumulate

combinations(items, r)      # choose r, order doesn't matter
permutations(items, r)      # choose r, order matters
product(a, b)               # cartesian product, replaces nested for-loops
list(accumulate(nums))      # running prefix sums in one call

accumulate over a manual prefix-sum loop

Prefix sums show up constantly (range-sum queries, subarray-sum problems) and the manual loop is boilerplate you can skip entirely.

✗ Manual
prefix = [0] * (len(nums) + 1)
for i, v in enumerate(nums):
    prefix[i + 1] = prefix[i] + v
✓ Idiomatic
from itertools import accumulate
prefix = [0] + list(accumulate(nums))

Both give you prefix[i] = sum of the first i elements, so sum(nums[a:b]) becomes prefix[b] - prefix[a] in O(1). The idiomatic version just removes a loop you'd otherwise write from scratch in nearly every prefix-sum problem.

Comprehensions, unpacking, and other reflexes

# comprehension over a manual loop + append
squares = [x * x for x in nums if x % 2 == 0]

# enumerate instead of range(len(...))
for i, val in enumerate(nums):
    ...

# zip to walk two sequences together
for a, b in zip(list1, list2):
    ...

# tuple/multiple assignment — no temp variable for swaps
left, right = right, left

# walrus operator — assign inside an expression (great inside while/if conditions)
while (line := stream.readline()):
    ...

# sort with a key instead of a custom comparator
items.sort(key=lambda x: (-x.priority, x.name))

# dict.get with a default instead of manual "if key in dict" checks
count = freq.get(ch, 0) + 1

Sorting with a composite key instead of a custom comparator

Tuple keys replace multi-level comparator functions

"Sort by priority descending, then by name ascending" is a two-level sort that people reach for a custom cmp function for — Python's key= handles it in one tuple, and negating a number flips only that level's direction.

✗ Comparator-style thinking
def compare(a, b):
    if a.priority != b.priority:
        return b.priority - a.priority   # descending
    return (a.name > b.name) - (a.name < b.name)  # ascending
# Python 3 has no cmp= for sort/sorted — this has to be
# wrapped in functools.cmp_to_key, which is slower and verbose.
✓ Idiomatic
items.sort(key=lambda x: (-x.priority, x.name))
# ties are broken left-to-right through the tuple, exactly like
# a multi-level comparator, with no cmp_to_key wrapper needed

Tuples compare element-by-element, so (-x.priority, x.name) sorts by priority descending first (negation flips numeric direction only), then by name ascending as the tiebreaker — the exact semantics of a two-level comparator, in one line. This trick only flips cleanly for numbers; to sort a string field descending, sort ascending and call .reverse(), or wrap it in a small helper — negating a string will error.

Building strings and checking membership at scale

''.join(...) over += in a loop

Strings are immutable in Python, so s += chunk inside a loop silently allocates a brand-new string every iteration — fine for a handful of iterations, a real O(n²) time bomb once n gets large (string-building problems, serializing a tree, etc.).

✗ O(n²) in the worst case
result = ""
for word in words:
    result += word + " "   # each += copies the ENTIRE string so far
✓ O(n)
result = " ".join(words)   # one allocation, not one per word

This is the string-building analogue of the "list as a queue" pitfall above: both look like a single operation but hide a full-copy cost. Some CPython versions optimize the simplest s += x loop internally, but it's implementation-specific — don't rely on an interpreter detail when str.join is guaranteed O(n) and just as short to write.

set for membership/intersection instead of list

"Have I seen this before?" and "what's common between these two collections?" are the two most common O(n) → O(1)/O(min(n,m)) upgrades available, and both are a one-word fix.

✗ O(n) per check
seen = []
for x in nums:
    if x in seen:        # O(n) scan through a list, every single call
        return True
    seen.append(x)
return False
✓ O(1) per check
seen = set()
for x in nums:
    if x in seen:        # O(1) average — hash lookup, not a scan
        return True
    seen.add(x)
return False

in means something completely different depending on the container: O(n) for list/tuple, O(1) average for set/dict. The same swap turns "find common elements between two arrays" from an O(n·m) nested loop into set(a) & set(b) at O(n+m).

None of this changes an algorithm's Big-O — it changes how fast and bug-free you write it under interview pressure, and interviewers explicitly notice fluent, idiomatic code as a signal.
